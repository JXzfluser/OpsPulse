# 真胶水：核心能力与使用极简

> **设计原则**：用户使用极简（三个动词、零侵入）；底层能力极重（契约、编排、闭环、策略）。  
> **一句话**：OpsPulse 不是多一个工具，而是让 **Issue、Agent、CI、组织规范** 四者说同一种语言。

---

## 1. 什么是「真胶水」

胶水层站在 **已有系统之间**，自己不变胖：

```
  人（TL/开发）          已有：GitHub Issue
        │                      │
        ▼                      ▼
  ┌─────────────────────────────────────┐
  │           OpsPulse（胶水）           │
  │  契约编译 │ 交付编排 │ 闭环写入 │ 策略  │
  └─────────────────────────────────────┘
        │              │              │
        ▼              ▼              ▼
  已有：Cursor      已有：GHA/Harness   已有：镜像仓/底座
  已有：业务代码    已有：JDK8 规范
```

| 真胶水 | 伪胶水（要避免） |
|--------|------------------|
| 业务仓 **0 文件** | 每仓复制 local-runner / mcp |
| 调你 **已有 CI** | 再造一套假流水线 |
| Issue 是 **唯一输入** | 还要学五份文档、八个脚本 |
| 能力在 **MCP 背后** | 能力摊给用户记 |

**价值前提**：团队痛点不是「不会写代码」，而是 **交付不可复现、Agent 不可控、验收不可追溯**。没有这个痛点，胶水层不必上。

---

## 2. 用户侧：极简表面（三个动词）

用户只需记住三个动作，对应三个 MCP Tool：

| 动词 | 工具 | 用户说什么 | 背后发生什么 |
|------|------|------------|--------------|
| **读单** | `parse_issue` | 「处理 Issue #45」 | 拉 Issue → 校验 → 输出可执行 Spec |
| **验货** | `trigger_pipeline` | 「跑 pr-validation」 | 编排 CI 阶段 → 汇总结果 |
| **结案** | `update_issue_status` | 「部署完，回写 Issue」 | 状态机 + 验收勾选 → Comment |

**配置一次性**：`GITHUB_PAT` + Cursor MCP。  
**业务仓**：打开 Cursor 写代码即可，**不安装 OpsPulse**。

---

## 3. 系统侧：四大核心能力（功能不能简单）

表面三个动词，背后四层「重」能力。这是真胶水与「脚本合集」的分水岭。

### 能力一：契约编译器（Issue → Executable Spec）

**不简单之处：**

- JSON Schema + 领域规则（bugfix 必带复现、JDK 必须 1.8、affected_paths 必填）
- `ready: false` **硬门禁** — Agent 不得开工
- 从自然语言正文 **抽取** 验收标准，与 frontmatter 对齐
- 单体仓多模块：一张 Issue 只绑定 **一个** `service` + `module_path`
- 输出 **稳定 JSON**，Agent 每次上下文一致（复利基础）

**用户无感**：只说「处理 #45」，拿到 `ready`、服务名、改哪些路径、AC-1/AC-2。

---

### 能力二：策略引擎（Org Policy → 执行约束）

**不简单之处：**

- JDK8 底座 **白名单**（企业镜像 vs 演示镜像）
- Dockerfile 模式校验：`FROM 底座` + `COPY jar`，禁止内嵌完整 JDK 下载
- `affected_paths` **写权限边界** — Agent 只能动工单声明范围
- 环境晋升规则：dev → staging → prod，免费/付费边界
- 与 `opspulse.yaml` / Issue frontmatter **合并优先级**（Issue 覆盖默认）

**用户无感**：违反策略时 parse 直接失败，并给出 **可修复** 的错误，而不是上线后爆。

---

### 能力三：交付编排器（Spec → Verified Artifact）

**不简单之处：**

- 三层模型编排：Layer1 底座校验 → Layer2 微服务 build → Layer3 服务镜像
- **适配** 多种 CI 后端：local-runner（自测）、GHA `workflow_dispatch`、Harness Remote Pipeline
- 阶段状态机：`success | failed | skipped`，失败 **分类**（Spec / 构建 / 镜像 / 冒烟）
- 与 PR / commit / issue number **关联** 可追溯
- 重试、超时、并行阶段（后续）

**用户无感**：一个 `trigger_pipeline`，看到阶段清单和红绿结果。

---

### 能力四：组织记忆写入（Delivery → Durable Record）

**不简单之处：**

- Issue 状态机：`parsed → in_progress → pr_open → validated → deployed`
- 验收项 **逐条映射** AC-id → passed/failed，不是一句「已完成」
- 结构化 Comment 模板，TL 可扫、可审计、可接 webhook
- FDE 回流：高频定制 → 模板 / Prompt / 流水线片段沉淀（长期复利）

**用户无感**：Issue 下多一条评论，勾选 ✅❌，工单即档案。

## 4. 能力深度 vs 使用简单（对照表）

| 维度 | 用户看到的 | 系统做的 |
|------|------------|----------|
| 输入 | Issue 号或一句话 | Schema、门禁、策略合并、模块路由 |
| 开发 | Cursor 正常改代码 | affected_paths 约束、Prompt 契约、PR 关联 |
| 验证 | 「验货」一次 | 多阶段 CI、三层镜像、失败归因 |
| 结案 | 一条 Comment | 状态机、验收矩阵、审计字段 |
| 多模块 | 换 Issue 不换配置 | service/module_path 路由到 monorepo 子路径 |
| 多仓库 | 同一套 MCP | Issue 内 repo 元数据 + 远程触发 CI |

## 5. 与「只用 Cursor」的差价（价值等式）

```
胶水价值 = （契约可执行 + 策略可强制 + 交付可验证 + 结果可审计）
         − （多出来的操作成本）
```

**操作成本必须 → 0**（相对 Cursor：多一句「处理 #45」）。  
**四项能力必须 → 明显强于** 自由对话 + 手动 mvn/docker。

若操作成本 > 多一句 Issue，产品失败 — 当前 MVP 的问题正在这里，不在概念。

## 6. 产品形态收敛（北向设计）

### 6.1 唯一主路径

```
GitHub Issue  →  「处理 #N」  →  Agent 改代码  →  「验货」  →  「结案」
```

禁止再主推：bootstrap 进业务仓、本地 e2e 当日常路径、手写大段 frontmatter。

### 6.2 辅助入口（降低契约编写成本）

| 入口 | 作用 |
|------|------|
| `new-issue.sh`（待做） | 问答式生成 Issue 正文，复制到 GitHub |
| GitHub Issue 模板 | 组织级一份，非每仓一份 |
| 示例 Issue | 仅文档参考 |

### 6.3 实现优先级（真胶水闭环）

1. **P0** `parse_issue` 从 GitHub 拉单 + chuanplus 实仓联调  
2. **P0** `trigger_pipeline` → GHA `workflow_dispatch`（真 CI，非 local-runner）  
3. **P1** `update_issue_status` 真实 post Comment（非 dry-run）  
4. **P1** 策略引擎：JDK 底座 / affected_paths 校验进 parse  
5. **P2** `opspulse handle #N` 单命令封装三动词  

`local-runner` **降级**为 OpsPulse 仓内开发自测，不对用户曝光。

## 7. 判定标准：是不是真胶水

上线前自问五条，有一条否就还不是：

- [ ] 业务仓是否 **零 OpsPulse 文件** 也能完整跑一单？  
- [ ] 用户是否只需 **三个动词** 就能描述全流程？  
- [ ] `ready=false` 是否 **挡住** Agent 乱开工？  
- [ ] 流水线是否打在 **客户已有 CI** 上（不仅是演示）？  
- [ ] Issue 结案是否留下 **可审计** 的验收记录？
