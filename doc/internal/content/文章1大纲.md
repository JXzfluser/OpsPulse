## 文章结构

### 1. 引子（300 字）

- 2026 年 Anthropic/OpenAI 都在复制 Palantir 的 FDE 模式
- 但 50–500 人 Java 团队请不起 FDE
- 你的日常：需求在 Issue 里、代码在 PR 里、上线在流水线里 — 三段割裂
- AI 工具只能补全一个文件，无法驱动全链路

### 2. 痛点场景（500 字）

**Before（现状）**：

```
产品经理提 Issue → 开发手动读需求 → Cursor 补全代码
  → 手动建 PR → 手动跑测试 → 手动部署 → 手动回写 Issue
```

- 每个环节信息丢失
- Issue 里的验收标准没人自动检查
- 流水线配置每次手写

**仓库锚点**：`examples/before-after.md`（Phase 2 创建）

### 3. 方案概览（400 字 + 架构图）

- OpsPulse 核心思路：**Issue = 交付契约**；**不是替代 Cursor，是套在外的交付流程层**
- 介入方式：模板 + MCP + 流水线 fork 进用户微服务仓库（详见 `doc/用户接入指南.md`）
- MCP 读 Issue → 解析为结构化 Spec → Agent 开发 → 流水线自动验证 → Issue 回写状态
- 架构图（复用 `doc/技术架构.md` 中的流程）

### 4. 核心设计 1：Issue Spec 契约（600 字）

- 为什么不用自由文本 Issue？
- YAML frontmatter 示例（精简版）
- `opspulse:auto` 标签 + `ready` 门禁
- **代码/配置锚点**：`schemas/issue-spec.v1.json`

### 5. 核心设计 2：MCP 三工具（500 字）

- `parse_issue`：把 Issue 变成机器可读的交付契约
- `trigger_pipeline`：一键触发验证/部署
- `update_issue_status`：回写验收结果到 Issue Comment
- **锚点**：`mcp-server/` 目录

### 6. 实战演示：30 分钟跑通（800 字 + 截图/录屏）

步骤：

1. `git clone` + 配置 PAT
2. 启动 MCP Server
3. 打开预置 Issue #1（order-service 微服务变更）
4. Cursor Agent：`parse_issue` → 改微服务代码/配置 → `trigger_pipeline`
5. 流水线：校验 JDK8 基础镜像 → build 微服务 → 打服务镜像 → 部署 → Issue 回写

（OpsPulse 仓内用 fixture 演示；真实 build 在用户微服务仓库执行）

**锚点**：`docs/quickstart.md`

### 7. 开源免费 vs 付费边界（300 字）

| 免费 | 付费 |
|------|------|
| Issue → PR → 测试通过 | 多环境晋升 |
| local-runner + GHA | 审批门禁 |
| Harness Dev 模板 | 合规扫描 |

**锚点**：`README.md` 对比表

### 8. FDE 沉淀：为什么这不是又一个 Demo（400 字）

- 虚拟 FDE：Issue/社群/咨询 = 前线嵌入
- 企业定制 48h 回流开源
- 产品化率 KPI

**锚点**：`doc/FDE方法论.md`
