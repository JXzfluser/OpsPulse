# OpsPulse FDE 方法论

> Forward Deployed Engineer（前线落地工程师）模式在独立开发者场景下的适配与实践。

---

## 1. FDE 概念对齐

### 行业 FDE（Palantir 原创，Anthropic/OpenAI 2026 复制）

| 原则 | 含义 |
|------|------|
| **嵌入一线** | 工程师驻场客户环境，参加 standup，直接交付 |
| **Day1 交付** | 拒绝需求文档表演；第一周必须有可运行代码 |
| **发现即工程** | 30–40% 时间用于客户对话，发现真实约束 |
| **土路铺高速** | 定制实现回流为平台通用功能 |
| **产品化率** | 成功指标 = 驻场交付中沉淀为产品的比例 |

### OpsPulse 虚拟 FDE（单人适配）

你无法真驻场，但通过以下方式等效：

| Palantir FDE | OpsPulse 虚拟 FDE |
|--------------|-------------------|
| 嵌入客户现场 | 嵌入 **GitHub Issue / 社群 / 咨询通话** |
| Day1 交付 | **每周 1 个可运行命令/模板** |
| 客户发现 | Agent 采集 + **每周 ≥2 次用户对话** |
| 土路铺高速 | 定制 → **48h 内抽象** → 开源子命令/模板 |
| 产品化率 KPI | ≥ 0.5 |

**OpsPulse 的 FDE 定义**：

> **Forward Discovery & Engineering** — 以虚拟前线嵌入持续发现 Java/DevOps 一线痛点，以 Day1 可运行交付验证价值，以土路铺高速将定制抽象为开源能力。

## 2. 三层资产模型（L1 / L2 / L3）

```
L3 企业一次性定制
  ↓ 48h 去标识 + 评估复用率
L2 可配置化（YAML 扩展 / Pro 插件）
  ↓ 2+ 客户共用
L1 开源标准化（主仓库）
```

| 层级 | 定义 | 存放位置 | 示例 |
|------|------|----------|------|
| **L1** | 80%+ 团队可直接复用 | `opspulse` 主仓库 | `parse_issue`、PR 验证流水线 |
| **L2** | 通过配置适配，无需改代码 | `opspulse.yaml` + Pro | 多环境变量、审批规则 YAML |
| **L3** | 客户环境特有、含私有信息 | 客户私有 fork | 客户内部域名、特有审批链 |

### 沉淀规则

- L3 **禁止**直接进入开源仓库
- L3 交付后 **48h 内**必须完成抽象评估
- 复用率 ≥ 80% → 升 L1；50–80% → 升 L2；< 50% → 保留私有，**不计入产品化率分子**

---

## 3. 产品化率 KPI

```
产品化率 = 过去 30 天升 L1/L2 的抽象资产数 / 过去 30 天 L3 交付次数
```

| 区间 | 动作 |
|------|------|
| ≥ 0.5 | 健康，可接更多定制 |
| 0.2 – 0.5 | 警告，减少定制比例 |
| < 0.2 连续 2 月 | **停止接纯定制**，专注开源迭代 |

### 计分示例

| 事件 | 产品化率影响 |
|------|--------------|
| 客户 A 审批门禁 → 抽象为 `stage-approval-manual.yaml` 进 L1 | 分子 +1 |
| 客户 B 内部 Jira 字段映射（不可复用） | 分母 +1，分子 +0 |
| 客户 C + D 共用多环境晋升模板 → 升 L2 | 分子 +1，分母 +2 |

## 4. 48h 产品化 SOP

### Day 0（交付日）

- 交付满足客户 Issue 契约的流水线/配置
- 记录 L3 交付到 `doc/需求池.md`

### Day 1（去标识）

- 剥离：域名、密钥、内部系统名、员工姓名
- 提取：通用逻辑、可参数化部分

### Day 2（评估 + 抽象）

| 复用率 | 动作 |
|--------|------|
| ≥ 80% | 提 PR 到开源主仓库（L1） |
| 50–80% | 写入 Pro 可配置模块（L2） |
| < 50% | 保留客户私有；复盘为何不可复用 |

## 5. 需求发现来源（虚拟前线嵌入渠道）

| 渠道 | 频率 | 产出 |
|------|------|------|
| GitHub Issue/Discussion | 每日 | 功能需求、Bug |
| 用户访谈 | 每周 ≥2 次 | 痛点原文、场景验证 |
| 公众号评论 | 每篇后 | 选题、定制线索 |
| 技术社群 | 每周 | 高频问题 |
| 咨询通话 | 按需 | L3 定制需求（强制走产品化 SOP） |

---

## 6. 与内容生产的绑定

**铁律**：每迭代 1 个功能，必须配套 1 篇公众号实战文章。

文章结构（永久复用）：

```
真实行业痛点
  → 通用 AI 工具解决不了的短板
  → 自研工具完整落地演示
  → 可复用脚本/模板下载
  → 文末话题收集需求
  → 引导 GitHub Star/Issue
```

---

## 7. 避坑清单（FDE 版）

1. **绝不跟风通用 AI 热点** — 失去垂直壁垒
2. **绝不做无沉淀的定制** — 变人力外包公司
3. **不盲目堆功能** — 每个功能必有文章案例
4. **不追求快速爆火** — 垂直复利 > 短期流量
5. **不依赖单一收入** — 开源 + 内容 + 私域 + 咨询四层兜底
6. **不跳过产品化评估** — L3 必须 48h 内处理
