# OpsPulse 技术架构

> MCP 调用逻辑、GitHub Issue 解析规则、Harness 流水线模板、本地 Runner 完整设计。  
> Phase 2 实现时直接参照本文档。

---

## 1. 架构总览

```
触发层        GitHub Issue（label: opspulse:auto）
              ↓
MCP 层        github-mcp-server（官方）+ opspulse-mcp-server（自研）
              ↓
契约层        Issue Spec v1 Schema → parse_issue → ready 判定
              ↓
开发层        Cursor Agent → branch → PR
              ↓
CI/CD 层      local-runner（默认）/ GHA / Harness（企业可选）
              ↓
回流层        update_issue_status → FDE 48h 抽象
```

## 2. MCP Server 拓扑

| Server | 来源 | 职责 |
|--------|------|------|
| `github` | [github/github-mcp-server](https://github.com/github/github-mcp-server) | 读 Issue/文件、建分支/PR、写 Comment |
| `opspulse` | 自研（Python FastMCP） | 解析契约、触发流水线、结构化状态回写 |

### `.cursor/mcp.json` 配置

```json
{
  "mcpServers": {
    "github": {
      "command": "docker",
      "args": [
        "run", "-i", "--rm",
        "-e", "GITHUB_PERSONAL_ACCESS_TOKEN",
        "ghcr.io/github/github-mcp-server"
      ],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_PAT}"
      }
    },
    "opspulse": {
      "command": "uv",
      "args": ["run", "opspulse-mcp"],
      "env": {
        "OPS_PULSE_CONFIG": "./opspulse.yaml"
      }
    }
  }
}
```

## 3. MCP 调用序列

```
1. github: search_issues（label:opspulse:auto state:open）
2. opspulse: parse_issue（owner, repo, issue_number）
   → IssueSpec + ready + errors
3. github: get_file_contents（affected_paths）
4. Cursor Agent：按 IssueSpec + prompts/issue-to-code.md 开发
5. github: create_branch → push_files → create_pull_request
6. opspulse: trigger_pipeline（pipeline=pr-validation, mode=local）
7. [通过] opspulse: trigger_pipeline（pipeline=deploy-dev）
8. opspulse: update_issue_status（state=deployed, acceptance_results）
9. github: add_issue_comment（结构化 Markdown）
```

## 4. opspulse-mcp-server 核心 Tool

### 4.1 `parse_issue`

**输入**：`owner`, `repo`, `issue_number`

**输出**：

```json
{
  "spec": { "type": "feature", "scope": "api", "..." },
  "raw_body": "...",
  "labels": ["opspulse:auto", "type:feature"],
  "errors": [],
  "ready": true
}
```

**解析优先级**：YAML frontmatter > 标签映射 > Issue 模板字段 > 正文启发式

### 4.2 `trigger_pipeline`

**输入**：

| 参数 | 类型 | 说明 |
|------|------|------|
| `pipeline_id` | enum | `pr-validation` / `deploy-dev` / `deploy-staging`（Pro） |
| `ref` | string | branch 或 PR head |
| `variables` | object | 透传 `harness.vars` |
| `mode` | enum | `local`（默认）/ `harness` / `github-actions` |

**输出**：`execution_id`, `status`, `stage_results[]`, `logs_url`

### 4.3 `update_issue_status`

**输入**：

| 参数 | 类型 |
|------|------|
| `state` | `parsed` / `in-dev` / `pr-open` / `testing` / `deployed` / `failed` |
| `pr_url` | string? |
| `pipeline_url` | string? |
| `acceptance_results` | `{ id, passed }[]` |

**输出**：Issue Comment Markdown（含验收清单勾选状态）

## 5. GitHub Issue 解析规则

### 5.0 交付模型（三层，必读）

```
┌─────────────────────────────────────────────────────────┐
│  Layer 1：JDK 1.8 基础镜像（企业统一提供，相对稳定）      │
│  例：registry.example.com/platform/jdk8-base:1.0         │
│  含：OS + JDK8 + 公共 agent/字体/证书 等运行时底座        │
│  ≠ 业务应用，≠ 整个微服务                                │
└─────────────────────────────────────────────────────────┘
                          ↓ FROM / 作为 CI 构建容器
┌─────────────────────────────────────────────────────────┐
│  Layer 2：微服务工程（用户业务仓库，Issue 驱动变更）      │
│  例：order-service、user-service（独立模块/仓库）          │
│  Agent 修改：src/**、pom.xml、application.yml 等          │
│  构建产物：jar/war（mvn/gradle，JDK8）                   │
└─────────────────────────────────────────────────────────┘
                          ↓ COPY artifact / 打服务镜像
┌─────────────────────────────────────────────────────────┐
│  Layer 3：服务镜像 / 部署单元（每次 Issue 可能变化）      │
│  例：registry.example.com/order-service:issue-45-abc     │
│  Dockerfile: FROM jdk8-base + COPY app.jar + 启动脚本    │
│  部署到：Dev/Staging/Prod                                │
└─────────────────────────────────────────────────────────┘
```

**OpsPulse 定位**：不替代 Layer 1（企业 JDK8 底座），不内置 Layer 2 完整微服务仓库（暂缓 sample-backend），提供 **Issue 契约 + 流水线模板 + MCP**，接入用户已有的微服务工程。

> 用户如何 fork 接入、与 Cursor 差异见 [用户接入指南.md](./用户接入指南.md)。

### 5.1 Issue Spec v1 Frontmatter

```yaml
---
opspulse_version: "1"
type: feature
scope: api
service:
  name: order-service              # 微服务名
  module_path: services/order-service/   # 在用户仓库中的路径
priority: P1
runtime:
  jdk_base_image: registry.example.com/platform/jdk8-base:1.0   # Layer 1
build:
  tool: maven                      # maven | gradle
  jdk: "1.8"
  command: mvn -pl services/order-service -am package -DskipTests
  artifact: services/order-service/target/order-service.jar
deploy:
  dockerfile: deploy/order-service/Dockerfile   # FROM ${jdk_base_image}
  image: registry.example.com/order-service     # Layer 3 推送目标
  tag_strategy: issue-{issue_number}-{git_short_sha}
  env: dev
harness:
  pipeline: pr-validation
  vars:
    JDK_BASE_IMAGE: registry.example.com/platform/jdk8-base:1.0
    SERVICE_NAME: order-service
acceptance:
  - id: AC-1
    given: "order-service 使用 JDK8 基础镜像构建并启动"
    then: "GET /actuator/health 返回 UP"
  - id: AC-2
    given: "本次 Issue 的 API 变更已部署 Dev"
    then: "验收用例 AC-2 通过"
affected_paths:
  - services/order-service/src/main/java/**
  - services/order-service/src/test/java/**
  - deploy/order-service/Dockerfile
---
```

**Schema 文件**：`schemas/issue-spec.v1.json`（JSON Schema Draft 2020-12）

### 5.2 标签 Fallback 规则

| 标签 | 映射 |
|------|------|
| `opspulse:auto` | **必须**，触发自动开发 |
| `type:*` | `spec.type` |
| `scope:*` | `spec.scope` |
| `component:*` | `service.name`（兼容旧标签） |
| `priority:*` | `spec.priority` |
| `pipeline:*` | `spec.harness.pipeline` |
| `env:*` | `spec.harness.env`（`staging`/`prod` 需 Pro） |

### 5.3 解析门禁（ready 判定）

| 条件 | 行为 |
|------|------|
| 缺 `opspulse:auto` | 不自动触发；仅手动调用 |
| frontmatter 格式错误 | `ready=false`，Comment 列 errors |
| `acceptance` 为空 | 警告；Agent 必须从正文提取 |
| `affected_paths` 为空 | 微服务类 Issue 建议必填；仅 infra/chore 可放宽 |
| `type=bugfix` 无复现步骤 | `ready=false`，拒绝自动开发 |

### 5.4 Issue 模板

- `.github/ISSUE_TEMPLATE/auto-dev-feature.yml`
- `.github/ISSUE_TEMPLATE/auto-dev-bugfix.yml`

自动标签：`opspulse:auto` + `type:feature` 或 `type:bugfix`

## 6. Harness 流水线模板

### 6.1 目录结构

```
.harness/
├── templates/
│   ├── stage-validate-spec.yaml
│   ├── stage-jdk-base-verify.yaml     # 校验/拉取 JDK8 基础镜像（Layer 1）
│   ├── stage-microservice-build.yaml  # 微服务 mvn/gradle 构建（Layer 2）
│   ├── stage-service-image-build.yaml # FROM jdk8-base + COPY jar（Layer 3）
│   ├── stage-deploy-dev.yaml
│   ├── stage-smoke-test.yaml
│   ├── stage-approval-manual.yaml     # Pro
│   └── stage-security-scan.yaml       # Pro
├── pipeline-pr-validation.yaml
├── pipeline-deploy-dev.yaml
└── pipeline-promote-staging.yaml
```

> **设计原则**：
> - **JDK8 基础镜像** = 运行时底座（Layer 1），流水线只**引用**不**重建**
> - **微服务** = 业务仓库中的独立服务（Layer 2），Issue 驱动代码变更后 **build 出 jar**
> - **服务镜像** = `FROM jdk8-base` + 业务 jar（Layer 3），每次 Issue 可能产出新 tag

同步副本：`harness-templates/`（开发用，CI 时复制到 `.harness/`）

### 6.2 `pipeline-pr-validation.yaml` 阶段

1. **Validate Issue Spec** — Schema + `service.name` + `runtime.jdk_base_image` 必填校验
2. **JDK Base Verify** — 拉取/校验 Layer 1 基础镜像可用（manifest、JDK 版本标签）
3. **Microservice Build** — 在 JDK8 环境（基础镜像作 CI 容器或宿主 JDK8）执行 `build.command`
4. **Service Image Build（可选于 PR 阶段）** — `docker build`：`FROM $JDK_BASE_IMAGE` + `COPY` artifact
5. **Smoke Test** — 临时起容器跑健康检查或单元测试报告门禁

### 6.3 `pipeline-deploy-dev.yaml` 阶段

1. **Build & Package** — 复用或执行微服务构建，产出 jar
2. **Push Service Image** — 构建 Layer 3 服务镜像并 push（tag 来自 Issue Spec）
3. **Deploy Dev** — 更新 Dev 环境 deployment（K8s/Compose）
4. **Smoke Test** — 对**已部署的微服务**做验收
5. **Update Issue** — `update_issue_status` + acceptance 勾选

### 6.4 Harness Git Connector 前置

- PAT Scope：`repo`, `admin:repo_hook`
- 启用 **API access**
- Remote Template 存储在仓库 `.harness/` 分支

## 7. 本地 Runner（默认 CI）

### 7.1 用法

```bash
./local-runner/run-pipeline.sh pr-validation --pr 12 --issue 45
./local-runner/run-pipeline.sh deploy-dev --issue 45
```

### 7.2 阶段对齐

| Harness 阶段 | local-runner 等价 |
|--------------|-------------------|
| validate_spec | `scripts/validate-issue-spec.py` |
| jdk_base_verify | `docker pull $JDK_BASE_IMAGE` |
| microservice_build | `mvn package`（用户仓库；MVP 可用 fixture jar 跳过） |
| service_image_build | `docker build -f deploy/Dockerfile` |
| deploy_dev | `docker compose up -d` / `kubectl set image` |
| smoke_test | `scripts/smoke-test.sh` |

### 7.3 本地演示环境（无 sample-backend）

OpsPulse 开源仓库**不包含**完整微服务工程（D11），演示分两层：

| 层级 | MVP 演示方式 |
|------|--------------|
| 契约 + MCP + 流水线 | `examples/issues/` + `examples/fixtures/`（模拟 order-service Issue） |
| 真实微服务 build | **接入用户自有微服务仓库**；或用 `examples/fixtures/app.jar` 占位跳过 build |

`deploy/Dockerfile` 示例（随模板提供）：

```dockerfile
ARG JDK_BASE_IMAGE=registry.example.com/platform/jdk8-base:1.0
FROM ${JDK_BASE_IMAGE}
COPY target/order-service.jar /app/app.jar
ENTRYPOINT ["java", "-jar", "/app/app.jar"]
```

本地可用公开 `eclipse-temurin:8-jre` 模拟 JDK8 基础镜像。
